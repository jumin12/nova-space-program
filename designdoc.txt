# Multiplayer Browser Space Program Game — Design & Architecture Doc

## 1. Working Title

**Project Codename:** *Orbital Frontier*

A multiplayer, browser-based, KSP-inspired space program simulator focused on realistic early rocketry, unmanned exploration, engineering constraints, science progression, missions, research, and cooperative/competitive space programs.

This is not a clone of KSP or KSP2. It should be a legally distinct game inspired by the broad genre of spaceflight construction and orbital simulation.

---

## 2. High-Level Vision

*Orbital Frontier* is a multiplayer space program game where players design, test, fail, research, and launch increasingly capable rockets from basic sounding rockets to early orbital spacecraft.

Unlike KSP’s relatively fast jump to crewed spaceflight, this game focuses heavily on the early game:

* Sounding rockets
* Uncrewed probes
* Telemetry and signal constraints
* Primitive guidance systems
* Reliability and part failure
* Atmospheric testing
* Suborbital milestones
* First satellite
* First lunar flyby
* First lunar impactor
* First lunar orbiter
* First soft landing
* Later crewed missions

The first major playable stage should cover everything equivalent to KSP’s early-to-mid tech progression up to roughly the **90 science tier**, but redesigned around a harder unmanned-first campaign.

---

## 3. Core Design Pillars

### 3.1 Build, Test, Fail, Improve

The player should not go directly from tiny rocket to orbit without testing. Early missions should reward partial success:

* Reach 5 km
* Reach 20 km
* Survive max-Q
* Transmit telemetry
* Separate a stage
* Recover instrument data
* Test an engine under certain conditions
* Achieve stable attitude control
* Complete first suborbital trajectory
* Reach space
* Reach orbit

Failure should still generate useful engineering data.

### 3.2 Unmanned First

The campaign should begin with probes, telemetry, primitive control, and ground-based science. Crewed missions should be locked behind substantial research, safety systems, life support, heat shielding, recovery systems, and proven launch reliability.

### 3.3 Multiplayer as a First-Class Feature

Players can share the same planet/moon system and space program world. Multiplayer should support:

* Cooperative space agencies
* Separate agencies on the same server
* Shared launch facilities
* Player-built craft visible to others
* Shared orbital environment
* Missions and contracts
* Race-to-milestone competition
* Spectating launches
* Rescue/recovery assistance

### 3.4 Browser-First Accessibility

The game runs in a modern browser using TypeScript, WebGPU/WebGL, and server-authoritative multiplayer. No install should be required for basic play.

### 3.5 Simulation Where It Matters

Use realistic-enough orbital and vehicle physics, but avoid trying to fully reproduce every edge case of a desktop physics sandbox. Split simulation modes:

* Rigid-body physics near ground/atmosphere/landing
* Orbital propagation for stable spaceflight
* Simplified part-joint simulation
* Server-authoritative state for multiplayer
* Client prediction only where safe

### Goal

Deliver the first complete playable version: one planet, one moon, VAB, research, missions, science, multiplayer, probes, early rockets, and progression up to roughly the KSP 90-science-equivalent tier.

This stage is the first real game.

### World Scope

* One Earth/Kerbin-like planet
* One Moon/Mun-like moon
* One primary launch site
* Optional secondary tracking stations as upgrades
* Atmosphere on home planet
* Vacuum moon
* Simplified terrain collision
* Basic biomes/science regions

### Gameplay Scope

* Vehicle Assembly Building

* Missions/contracts
* Science collection and transmission
* Funding/reputation or program points
* Probe-first progression
* Early telemetry and communications limits
* Launch vehicle design
* Sounding rocket campaigns
* First satellite
* First lunar missions
* Multiplayer agency support

### Stage 1 Progression Theme

The player starts with technology that is barely space-capable:

* No crewed capsules

## 5.1 Vehicle Assembly Building

### Core VAB Requirements

The VAB must support:

* Part list grouped by category
* Stack attachment
* Radial attachment
* Symmetry modes
* Staging editor
* Center of mass indicator
* Center of thrust indicator
* Center of pressure indicator, once aerodynamics research is unlocked
* Delta-v estimate, initially locked or approximate
* Thrust-to-weight estimate
* Mass, cost, reliability, and power estimates
* Save/load craft files
* Share craft with agency/server
* Multiplayer collaborative editing later

### Part Categories

* Command and control
* Probe cores
* Fuel tanks
* Engines
* Solid motors
* Structural parts
* Decouplers
* Aerodynamics
* Science
* Power
* Communication
* Recovery
* Utility

### Early VAB Limitations

The early game should intentionally restrict the player:

* Maximum vehicle mass
* Maximum part count
* Maximum height
* Limited staging complexity
* Poor construction tolerances
* Limited test data means some part stats are approximate

Facility upgrades can improve these limits.

## 5.2 Research and Science

### Design Goal

Research should feel like an engineering program, not just collecting points from biomes. The player earns science/engineering data from tests, flights, telemetry, recovered instruments, and milestone completion.

### Science Sources

* Ground tests
* Static fire tests
* Atmospheric telemetry
* High-altitude telemetry
* Space environment measurements
* Recovered payloads
* Lunar flyby data
* Lunar impact data
* Lunar orbital scans
* First soft-landing data
* Mission milestone bonuses

### Data Types

Instead of one generic science currency, Stage 1 may use either one combined currency or several lightweight categories:

* **Engineering Data:** engines, structures, staging, reliability
* **Flight Data:** aerodynamics, control, stability
* **Space Science:** upper atmosphere, vacuum, radiation, lunar studies
* **Comms Data:** antennas, telemetry, signal processing

For the first implementation, a single science currency is acceptable, but the UI should be designed so typed data can be added later.

Missions should guide players through real engineering history-like steps. They should not just say “go to orbit.” They should create pressure to solve specific problems.

### Mission Categories

* First launch
* First recovered payload
* First spaceflight
* First orbit
* First satellite
* First lunar flyby

* First lunar orbit
* First lunar landing

#### Engineering Tests

Repeatable or semi-repeatable missions:

* Test engine at altitude
* Test decoupler under acceleration
* Static fire a new engine
* Demonstrate stable spin-stabilized flight
* Demonstrate guided pitch-over
* Demonstrate controlled reentry

#### Science Missions

* Measure lower atmosphere
* Measure upper atmosphere
* Measure vacuum radiation
* Scan planet from polar orbit
* Return high-altitude samples
* Transmit lunar flyby data

#### Commercial/Agency Contracts

* Launch a small payload to suborbital trajectory
* Place satellite into specific orbit
* Deploy relay satellite
* Impact probe on moon
* Deliver surface science package

#### Multiplayer Missions

* Joint relay network
* Shared station precursor target
* Competitive first-to-orbit milestone
* Cooperative lunar mapping campaign
* Rescue stranded probe or recover debris

## 5.5 Difficulty and Early Game Hardness

### Core Differences from KSP

The early game should be harder and more structured.

Restrictions:

* No starting crewed pod
* No strong reaction wheels at start
* No perfect SAS at start
* Limited antenna range
* Primitive probes have limited command capability
* Batteries are heavy and weak
* Solar power unlocks later
* Engines have reliability ratings
* Staging can fail if parts are untested or low quality
* Aerodynamic instability is a major early challenge
* Science instruments require power, storage, and/or transmission
* Reentry survival is not guaranteed

### Flight Computer Progression

Early probes should not behave like perfect pilots.

Possible command levels:

1. **Unguided:** launch only
2. **Timer Commands:** stage after X seconds, pitch after X seconds
3. **Primitive Stability:** hold approximate attitude
4. **Programmed Guidance:** execute pitch program
5. **Orbital Guidance:** hold prograde/retrograde
6. **Maneuver Execution:** execute planned burns
7. **Advanced Autonomy:** handle signal delay and scripted mission events

### Communication Constraints

Early communications should matter.

* Antenna range limits
* Line-of-sight requirement optional in Stage 1, recommended later
* Data rate limits
* Signal blackout during reentry
* Probe command lockout when out of range, depending on difficulty
* Relay satellites become important before lunar missions

# 6. Physics Design

## 6.1 Simulation Modes

The game should use multiple physics modes rather than one universal simulation.

### Mode A — Active Rigid-Body Flight

Used for:

* Launch
* Atmospheric flight
* Landing
* Crashes
* Docking/rendezvous proximity
* Surface operations

Technology:

* Rapier.js or custom rigid-body layer
* Fixed timestep
* Simplified part-joint constraints
* Server authority for multiplayer

### Mode B — Orbital On-Rails Simulation

Used for:

* Stable orbit
* Time warp
* Long-duration coast
* Unloaded vessels
* Satellites
* Lunar transfers

Technology:

* Custom TypeScript orbital mechanics module
* Patched conics for Stage 1
* Keplerian orbit propagation
* Sphere of influence transitions
* Deterministic server-side propagation

### Mode C — Simplified Background Simulation

Used for:

* Remote probes
* Relay satellites
* Long-term missions
* Mission timers
* Power drain during coast

This should simulate:

* Orbit position over time
* Battery charge/drain
* Solar generation
* Communication status
* Science data transmission
* Critical event timers

---

## 6.2 Physics Responsibilities

### Client

* Render prediction
* Local input collection
* Camera and UI
* Temporary visual effects
* Smooth interpolation
* Non-authoritative simulation preview in VAB

### Server

* Authoritative vessel state
* Mission validation
* Orbital propagation
* Collision outcome authority
* Science and reward validation
* Multiplayer state sync
* Persistence

## 6.3 Rocket Part Physics

### Stage 1 Simplification

Do not simulate every part as a fully independent rigid body all the time.

Recommended approach:

* Treat vessel as a tree of parts
* Compute aggregate mass, center of mass, moment of inertia, thrust vectors, drag profile
* Use simplified flex/joint stress model
* Break parts when stress exceeds limits
* Use rigid-body physics for whole vessel or major separated assemblies

This is much more browser-friendly than simulating hundreds of constrained rigid bodies.

### Part Failure Model

Each part has:

* Reliability rating
* Tested/unverified status
* Max temperature
* Max dynamic pressure
* Max acceleration
* Max joint stress
* Ignition reliability
* Burn duration tolerance

Failures can include:

* Engine fails to ignite
* Engine underperforms
* Engine explodes
* Decoupler fails
* Antenna breaks
* Battery overheats
* Parachute rips off
* Structural joint failure

Failure chance should be tunable by difficulty.

---

## 6.4 Aerodynamics

Stage 1 should use simplified but meaningful aerodynamics.

Model:

* Drag coefficient per part
* Cross-section area approximation
* Center of pressure calculation
* Atmospheric density curve
* Mach effects simplified
* Max-Q stress
* Fin stabilization
* Nose cone drag reduction

Important gameplay outputs:

* Rockets flip if badly designed
* Long skinny rockets need fins/control
* Fairings matter for satellites
* Max-Q can destroy fragile vehicles

---

## 6.5 Orbital Mechanics

Stage 1 should use patched conics, not full n-body.

Required:

* Position and velocity vectors
* Apoapsis/periapsis
* Inclination
* Time to apoapsis/periapsis
* Escape trajectory
* Moon encounter prediction
* Orbit lines
* Maneuver nodes after research unlock
* SOI transitions

Time warp should only run on the orbital solver, not full rigid-body physics.

# 7. Multiplayer Design

## 7.1 Multiplayer Modes

### Cooperative Agency

Players share:

* Funding
* Science
* Tech tree
* Facilities
* Craft library
* Mission progress
* Relay network

### Competitive Agencies

Players share the same celestial system but have separate:

* Funding
* Science
* Tech tree
* Missions
* Launch schedules
* Craft libraries

### Sandbox Server

All parts unlocked. Focus on building, launching, and playing together.

---

## 7.2 Multiplayer Time Warp Problem

Time warp is the hardest multiplayer design issue.

### Stage 1 Recommendation

Use **agency/world time with scheduled warp votes** or **per-vessel mission instances**.

Options:

#### Option A — Shared World Time

* Everyone exists on the same timeline
* Time warp requires agreement or admin control
* Good for realism
* Bad if many players want independent missions

#### Option B — Mission Instances

* Active launch happens in a mission instance
* Orbital results are committed to shared world after mission completion
* Easier for multiplayer
* Less seamless

#### Option C — Hybrid

Recommended.

* Launches and active flights occur in real time or controlled local warp
* Stable orbits are propagated analytically on the server
* Players can request time acceleration for their own mission if it does not conflict with other active events
* Server resolves event order by universal timestamp

Stage 1 should implement the hybrid model.

## 7.3 Network Architecture

### Server Authoritative Model

The server owns:

* Vessel state
* Physics truth
* Mission success
* Science rewards
* Collisions and destruction
* Orbit transitions
* Resource consumption

Clients send:

* Control inputs
* Staging commands
* VAB changes
* Mission selections
* Camera/spectator preferences

Server broadcasts:

* Vessel transforms
* Telemetry
* Events
* Explosions/destruction
* Orbit updates
* Player presence

### Recommended Multiplayer Stack

* Node.js
* TypeScript
* Colyseus for rooms/state sync
* uWebSockets.js for performance later if needed
* Redis for ephemeral state/cache
* PostgreSQL for persistence

# 16. Major Risks

## 16.1 Scope Creep

A full KSP2-like game is enormous. Stage 1 must stay focused on one planet, one moon, early tech, and unmanned progression.

## 16.2 Multiplayer Time Warp

This must be designed early. Do not bolt multiplayer onto a single-player time system later.

## 16.3 Physics Complexity

Avoid simulating every part as a full rigid body unless absolutely necessary. Use aggregate vessel physics and simplified joint stress.

## 16.4 Browser Performance

Limit part counts, simplify terrain collision, and keep active physics objects low.

## 16.5 Legal/IP Risk

Do not use KSP names, parts, planets, Kerbals, UI layout, art style, branding, or exact tech tree. Make a distinct game in the spaceflight construction genre.

# 19.1 Physics Engine Architecture Spec

## Goal

Build a browser-capable physics system that feels similar to KSP-style part-based rockets:

* each part has mass and forces
* joints flex and can break
* poor designs wobble
* staging physically separates craft
* debris persists
* aero forces affect individual parts
* engines apply thrust at real positions
* asymmetric thrust causes torque

The physics should prioritize believable, emergent behavior over perfect real-world simulation.

---

## Physics Engine Layers

### Layer A — Math Core

Shared deterministic math utilities:

* Vec3
* Quat
* Mat3
* Transform
* inertia tensor helpers
* orbital element conversions
* fixed timestep utilities

All physics code should use the same math package.

---

### Layer B — Part Runtime

Each part is represented at runtime by:

```ts
type RuntimePart = {
  id: string;
  definitionId: string;
  parentId?: string;
  childIds: string[];

  localTransform: Transform;
  worldTransform: Transform;

  mass: number;
  dryMass: number;
  resourceMass: number;
  inertiaTensor: Mat3;

  temperature: number;
  damage: number;
  destroyed: boolean;

  resources: ResourceContainer;
  modules: RuntimePartModule[];

  rigidBodyHandle?: number;
  colliderHandles: number[];
  jointHandles: number[];
};
```

---

### Layer C — Vessel Runtime

A vessel contains:

```ts
type RuntimeVessel = {
  id: string;
  ownerPlayerId?: string;
  agencyId: string;

  parts: Map<string, RuntimePart>;
  rootPartId: string;
  stages: RuntimeStage[];

  simulationMode: "active-rigid" | "packed-orbit" | "landed" | "destroyed";

  bodyId: string;
  referenceFrame: ReferenceFrame;

  centerOfMass: Vec3;
  totalMass: number;
  aggregateInertia: Mat3;

  orbit?: OrbitState;
  timeline: VesselTimelineState;
};
```

---

### Layer D — Active Rigid Physics

Used for:

* launch
* atmospheric flight
* landing
* docking proximity
* collisions
* nearby multiplayer
* explosions

Implementation options:

1. Rapier.js/WASM for rigid bodies and constraints
2. Custom TypeScript constraint solver for vessels only
3. Hybrid: Rapier for collisions, custom joints for rocket flex

Recommended initial implementation:

* Rapier for rigid bodies, colliders, broadphase, contact resolution
* custom vessel joint stress layer on top
* custom aero and engine forces

---

### Layer E — Packed Orbital Physics

Used for:

* stable orbits
* high time warp
* unloaded vessels
* long coasts
* background satellites

Packed vessels do not simulate every part as active rigid bodies.

They preserve:

* part state
* resources
* power
* science data
* orbit
* scheduled events

They do not solve:

* per-frame joint wobble
* collision contact
* aerodynamic forces unless reentering

---

## Active Physics Step Order

Each fixed physics tick:

1. Read player/server input commands.
2. Apply staging commands.
3. Update resource flow.
4. Update engine thrust.
5. Apply thrust forces at engine positions.
6. Apply gravity to all bodies.
7. Calculate atmosphere density at each part.
8. Apply aero drag/lift/heating per part.
9. Update thermal conduction and radiation.
10. Solve rigid-body physics.
11. Evaluate joint stress and flex.
12. Break failed joints.
13. Split disconnected vessel graphs.
14. Detect collisions and impacts.
15. Apply damage/explosions.
16. Update telemetry/science/mission events.
17. Emit authoritative snapshot.

---

## Fixed Timestep

Recommended:

* physics tick: 30 Hz server baseline
* optional 60 Hz for local single-player or small active scenes
* render interpolation: client-side
* high-load fallback: reduce solver iterations, not tick consistency

Example:

```ts
const FIXED_DT = 1 / 30;
```

---

## Joint Model

Each attachment joint has:

```ts
type RuntimeJoint = {
  parentPartId: string;
  childPartId: string;

  anchorParent: Vec3;
  anchorChild: Vec3;

  linearStiffness: number;
  angularStiffness: number;
  damping: number;

  breakForce: number;
  breakTorque: number;

  accumulatedStress: number;
  failed: boolean;
};
```

### Wobble Behavior

To create KSP-like wobble:

* joints should not be infinitely stiff
* tall rockets should bend
* heavy payloads should stress lower joints
* radial boosters should flex
* struts should reduce flex

### Joint Failure

A joint fails when:

* force exceeds breakForce
* torque exceeds breakTorque
* accumulated fatigue exceeds threshold
* temperature weakens joint below load
* collision shock exceeds structural rating

---

## Engine Force Model

Each engine applies force at its nozzle transform.

```ts
force = thrustDirection * currentThrust;
torque = cross(enginePosition - centerOfMass, force);
```

Engine state includes:

* throttle
* ignition state
* fuel availability
* atmosphere pressure
* gimbal angle
* reliability
* heat output

Failures:

* no ignition
* hard start explosion
* thrust loss
* stuck throttle
* gimbal lock
* overheating

---

## Aero Force Model

Per part:

```txt
q = 0.5 * density * velocity^2
Drag = q * Cd * area
Lift = q * Cl * area
```

Each part has:

* drag coefficient
* lift coefficient
* exposed area
* occlusion/shielding state
* max dynamic pressure

Fairings and cargo bays shield enclosed parts.

---

## Damage Model

Damage sources:

* impact
* overheating
* overpressure
* aerodynamic stress
* joint stress
* engine explosion

Damage effects:

* part destroyed
* resource leak
* reduced engine thrust
* antenna disabled
* solar panel disabled
* joint weakened

---

## Vessel Splitting

When joints break or decouplers fire:

1. Build graph of connected parts.
2. Find connected components.
3. Create one RuntimeVessel per component.
4. Transfer velocities and angular velocities.
5. Assign ownership and vessel type.
6. Mark debris if no command/control part exists.

---

# 19.2 Netcode and Determinism Spec

## Core Rule

The server is authoritative.

Clients never decide:

* mission completion
* science rewards
* collisions
* destruction
* orbit commits
* resource truth
* other players’ vessel states

---

## Networking Model

### Client Sends Commands

```ts
type ClientCommand =
  | { type: "setThrottle"; value: number; tick: number }
  | { type: "setAttitudeInput"; pitch: number; yaw: number; roll: number; tick: number }
  | { type: "stage"; tick: number }
  | { type: "toggleSAS"; enabled: boolean; tick: number }
  | { type: "activatePart"; partId: string; action: string; tick: number }
  | { type: "requestWarp"; warpRate: number };
```

### Server Sends Snapshots

```ts
type VesselSnapshot = {
  serverTick: number;
  vesselId: string;
  mode: VesselSimulationMode;
  bodyId: string;
  partTransforms?: PartTransformSnapshot[];
  aggregateTransform?: Transform;
  velocity: Vec3;
  angularVelocity: Vec3;
  resources: ResourceSnapshot;
  events: ServerEvent[];
};
```

---

## Snapshot Rates

Recommended:

* active nearby vessel: 15–30 snapshots/sec
* distant active vessel: 5–10 snapshots/sec
* packed orbital vessel: event-based or 1/sec
* map-only vessel: orbit elements only

---

## Client Prediction

Clients may predict:

* local input feel
* throttle response
* camera movement
* UI readouts
* vessel motion between snapshots

Clients should not permanently commit predicted results.

---

## Reconciliation

When authoritative snapshot arrives:

1. Compare predicted state to server state.
2. If error is small, smooth correction.
3. If error is large, snap or replay recent inputs.
4. Always trust server for destruction/failure events.

---

## Determinism Strategy

Full deterministic physics across browsers is not guaranteed.

Therefore:

* do not rely on lockstep multiplayer physics
* server runs truth simulation
* clients interpolate
* replays store commands plus authoritative checkpoints
* important results are event-sourced

---

## Multiplayer Time Warp Determinism

Time warp is handled analytically by server.

The server stores:

```ts
type VesselTimelineState = {
  localTime: number;
  universalTimeAnchor: number;
  warpRate: number;
  domainId: string;
  canWarp: boolean;
};
```

Vessels in different warp domains may advance independently.

When domains merge:

* future events are resolved by timestamp
* vessels are propagated to common sync time
* active physics begins only after synchronization

---

## Anti-Cheat

Server validates:

* available fuel
* throttle limits
* staging order
* unlocked parts
* craft mass/cost
* mission requirements
* science validity
* warp permissions

# 19.5 WASM / Rust Integration Plan

## Why WASM

Heavy physics and orbital math may exceed comfortable JS performance.

Use WASM for:

* constraint solving
* collision acceleration
* orbital propagation batches
* terrain collision queries
* large matrix/vector operations

---

## Stage 1 Approach

Start with TypeScript plus Rapier.js.

Only add custom Rust/WASM once bottlenecks are measured.

---

## Candidate WASM Modules

### physics_core_wasm

Responsibilities:

* joint solver
* stress calculation
* rigid cluster updates
* force integration

### orbit_core_wasm

Responsibilities:

* propagate many orbits
* SOI prediction
* maneuver solving
* encounter search

### terrain_query_wasm

Responsibilities:

* height sampling
* raycasts
* collision queries
* biome lookup

---

## Rust API Shape

```ts
type WasmPhysicsInput = {
  partBuffer: Float64Array;
  jointBuffer: Float64Array;
  forceBuffer: Float64Array;
  dt: number;
};
```

Keep data transfer minimal.

Avoid passing thousands of JS objects across the WASM boundary.

---

## WASM Rules

* transfer typed arrays, not objects
* batch calls
* avoid per-part JS/WASM calls
* keep server and client module versions identical
* store deterministic checkpoints

# 19.10 Final Engineering Warning

The full design is technically possible in JavaScript/TypeScript, but the risky parts are:

* KSP-like joint physics
* browser performance with many active parts
* multiplayer time warp
* server cost for active physics
* save/replay consistency
* floating-origin bugs
* terrain collision at planet scale

The safest development strategy is:

1. Build single-player physics first.
2. Add server-authoritative multiplayer early.
3. Keep Stage 1 part counts controlled.
4. Use orbital rails aggressively outside physics bubbles.
5. Measure performance before adding WASM.
6. Make the game fun before making it huge.

The project should be treated as a serious engine/game development effort, not just a normal web app.

# 20.5 Nanite-Style Geometry Streaming Spec

## Goal

Support extremely detailed static terrain/objects later without overwhelming the browser.

This system is optional for Stage 1 and should not block core gameplay.

---

## Use Cases

Good candidates:

* mountains
* cliffs
* crater rims
* launchpad structures
* rocks
* anomalies
* static colonies later

Bad candidates:

* active rocket parts
* physics colliders
* moving vessels
* tiny debris
* UI objects

---

## Virtual Geometry Principles

The renderer streams clusters of triangles based on screen-space error.

Each cluster stores:

* bounding sphere/box
* geometric error
* material id
* vertex/index data
* parent/child hierarchy

---

## Cluster Pipeline

Offline asset process:

1. Import high-poly mesh.
2. Split into meshlets/clusters.
3. Build hierarchy.
4. Generate simplified parent clusters.
5. Compress cluster data.
6. Store streaming pages.

Runtime:

1. Determine visible clusters.
2. Request needed pages.
3. GPU cull clusters.
4. Render selected clusters.
5. Evict unused pages.

---

## Browser Constraints

Because this is browser-based:

* keep cluster sizes moderate
* stream compressed binary chunks
* avoid huge CPU decompression spikes
* use workers for decoding
* keep GPU memory budget strict

---

## Collision Separation

Nanite-style geometry is visual only.

Collision uses simplified proxy geometry.

# 20.13 Rendering Risk Notes

Do not build the Nanite-style renderer first.

The correct order is:

1. Make rockets launch.
2. Make orbit work.
3. Make multiplayer work.
4. Make terrain readable.
5. Add shaders and visual polish.
6. Add virtualized geometry only after performance budgets are understood.

A beautiful renderer cannot save an unplayable flight model.

The visual goal is not to beat AAA games. The goal is to make a browser-based space program simulator that is readable, performant, and inspiring.
